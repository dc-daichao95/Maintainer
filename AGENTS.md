# 项目：Kernel-Maintainer

基于开源的 Sashiko 项目构建，提供专业的内核代码检视（Kernel Code Review）Agent。通过整合自定义的内核检视规则与友好的 Web UI 界面，实现对内核源码库的高效自动化审查与风险排查。

## 核心准则 (Core Guidelines)
作为参与本代码库开发的 AI 智能体，你必须遵循以下底线原则：

1. **先思考后编码 (Think Before Coding)**：不要臆测。明确说明假设。如果有疑问，停下来提问。
2. **简单优先 (Simplicity First)**：编写能解决问题的最少代码。不添加推测性的功能。
3. **外科手术式修改 (Surgical Changes)**：只修改必须修改的地方。不要在未经要求的情况下“改进”相邻的代码。
4. **目标导向执行 (Goal-Driven Execution)**：定义可验证的成功标准。将命令式任务转化为可验证的目标（如测试），循环执行直至成功。
5. **保证与上游持续集成 (Ensure Upstream Continuous Integration)**：**【核心狠心原则】** 任何新提出的方案、架构设计或代码变更，必须将“能够与 `sashiko` 开源项目主干保持持续集成与平滑升级”作为第一考量。绝不能为了实现特性而采取破坏性、无法向后兼容或无法合并上游更新的飞线设计 (Hack)！
6. **尽量不修改原生代码 (Minimize Native Code Modifications)**：**【强制约束】** 本项目尽量不修改 `sashiko` 原生代码，只在 `my-src` 目录下添加相关的功能。如果涉及原生代码的修改，在设计和开发阶段必须停下，得到人类的同意后方可继续。

## 设计原则 (Design Principles)
1. **严禁直接写示例代码**
2. **配合 UML 图阐述**：通过 PlantUML 画 UML 图配合说明阐述设计逻辑。在界面设计环节，可以使用 PlantUML Salt 绘制线框图 (Wireframe) 进行前端设计 Demo。
3. **动静结合**：设计时注重动态的功能设计，以及静态的软件架构设计。
4. **贴合内核场景**：检视规则和 UI 呈现必须考虑内核代码的特殊性（如内存分配、并发原语、中断上下文等）。

## 技术栈 (Stack)
- **语言**：Rust
- **核心引擎**：基于开源 `sashiko` 框架
- **Web 框架**：Axum / Actix-Web, Tera / Askama (模板), TailwindCSS (CDN) 等 UI 组件
- **数据库**：基于 SeaORM / Diesel (ORM) 的 SQLite
- **检视后端**：自定义内核检视规则解析器（结合正则、AST 分析及 `sashiko` 核心能力）
- **测试**：`cargo test`
- **报告**：JSON, 文本 (Text), Excel

## 目录结构 (Directory Structure)
本项目基于开源 `sashiko` 框架演进，为保持与上游项目的平滑同步，我们将项目原生内容与本项目的自定义内容进行了明确的目录隔离：

- `my-src/`：**【核心关注】** 本项目自定义的 Rust 后端源码、产品设计文档、检视规则以及前端 Web UI 代码等全部存放在此目录下。所有的定制化开发必须在此目录下进行。
- 其他目录：属于开源 `sashiko` 项目原生代码或基础设施。除非涉及核心引擎 Bug 修复或必要的能力扩展集成，原则上避免在这些原生目录中硬编码本项目的特定业务逻辑，以便后续与上游版本同步更新。

## 代码质量与风格 (Code Quality & Style)
本项目的代码必须达到高质量、高可读性并符合社区规范，智能体在提交任何代码前必须严格检查以下标准：

1. **命名规范**：
   - 变量、函数与模块名：使用小写字母及下划线 (`snake_case`)。变量命名必须具备高度可读性，能清晰表达其用途。
   - 结构体、枚举与 trait 名：使用大驼峰式 (`PascalCase`)。
   - 常量与静态变量：使用全大写及下划线 (`UPPER_SNAKE_CASE`)。
2. **长度与规模限制**：
   - 函数长度：单个函数不应超过 50 行。
   - 文件长度：单个文件不应超过 500 行。
3. **注释与文档**：
   - 对于非直观的业务逻辑、复杂的正则或算法，必须添加精确的行内注释解释“为什么(Why)”这样做，而不是“做了什么(What)”。
   - 所有公开的结构体、模块、方法和函数，都应包含标准格式的 Rustdoc (`///`)。
   - 避免无意义的自明注释。
4. **防范复杂度**：
   - 函数应当短小精悍，遵循单一职责原则 (Single Responsibility Principle)。
   - 控制代码嵌套层级（禁止超过4层嵌套），尽量采用提早返回 (Early Return) 结合 `?` 操作符处理错误，避免深层的 `if/else` 或 `match` 嵌套。
5. **容错与错误处理**：
   - 绝不滥用 `unwrap()` 或 `expect()`，除非有绝对把握该处不可能 panics（并在注释中说明理由）。
   - 使用自定义错误类型（如基于 `thiserror` 或 `anyhow`）传递清晰的错误上下文。向用户输出错误信息时提供可操作的建议。

## 约定 (Conventions)
- **类型与所有权 (Types & Ownership)**：严格遵守 Rust 的类型系统与所有权规则，合理使用生命周期与借用检查，避免不必要的 `clone()`。
- **状态管理**：在 Web 框架中，合理使用状态扩展（如 `Extension` 或 `State`）安全地跨线程管理数据库连接池和外部应用状态。
- **架构分层**：严格保持核心检视逻辑（规则解析/文件处理管道）与 Web/UI 层解耦。业务逻辑不应当混入路由控制器中。

## 边界 (Boundaries)
- **范围限制**：绝对不要重构用户请求的直接范围之外的代码。
- **死代码**：由于 Rust 编译器会给出 `dead_code` 警告，修改导致未使用的导入或变量时，必须删除它们或修复警告。
- **基础设施**：除非特别指示，否则不要修改 CI/CD 管道 (`.github/workflows`) 或核心检视规则配置。

## 验证与完成 (Verification & Completion)
在完成任务前：
1. **测试**：必须运行 `cargo test` 并确保 0 个失败。
2. **覆盖率**：新增功能必须被新测试覆盖。
3. **可追溯性**：每一行更改都必须能直接追溯到用户的请求。严禁顺手牵羊式的重构 (No drive-by refactoring)。
4. **架构验证**：在合并或完成任务前，必须验证代码更改是否符合技术栈约定与分层架构（如 Web/API 层与核心检视逻辑必须保持解耦）。绝不引入破坏现有架构模式的跨层依赖。

## 子智能体目录 (Subagents Directory)
本仓库和 Cursor 环境支持以下子智能体 (Subagents) 以协助不同类型的任务。请根据任务类型主动唤起它们（例如：`@tech-leader`）：

- **@requirements-analyst**：需求分析专家。在开始新功能前调用，负责通过与人类对话进行脑暴，探讨需求边界并分析需求范围。
- **@feature-designer**：特性设计专家。在脑暴阶段完成后调用，根据访谈内容生成特性设计文档 (Spec) 并同步完成测试策略设计。
- **@planner**：需求分析与拆解专家。在开始新功能前调用，用于拆解任务、制定执行计划和 Checklists。
- **@ui-designer**：UI/UX 设计专家。负责根据指定的设计风格和规范进行网页界面的设计和美化。
- **@developer**：研发专家。在完成需求规划后调用，负责严格按照 TDD 流程编写测试用例并提供代码实现，处理审查反馈。
- **@tech-leader**：技术负责人与团队编排者。当需要规划、分配任务、跨阶段协调多个智能体（如开发、测试、审查）或处理复杂的全局项目流程时，务必调用此智能体。
- **@rust-reviewer**：后端代码审查专家。在开发完成后调用，重点负责后端 Rust 代码的审查，把关代码质量与测试独立性。
- **@frontend-code-reviewer**：前端代码审查专家。在前端开发完成后调用，重点检视前端代码问题，防范页面崩溃和死循环。
- **@qa**：端到端质量保证与 Web 自动化专家。在代码审查无意见后调用，负责通过 E2E 测试用例及基于浏览器的自动化测试保证代码整体运行正确。
- **@generalPurpose**：通用智能体。用于研究复杂问题、搜索代码或执行多步任务。当你需要搜索某个关键字或文件，但不确定能快速找到时使用。
- **@explore**：专用于探索代码库的只读智能体。速度快，当你需要按模式查找文件、搜索代码关键字或询问有关代码结构机制的问题时使用。
- **@shell**：命令行执行专家。用于运行 Git 操作、命令执行及其他终端任务。
- **@cursor-guide**：Cursor 产品文档向导。当需要了解 Cursor Desktop、IDE、CLI 机制或功能使用问题时调用。
- **@best-of-n-runner**：在隔离的 Git 工作树中运行任务。用于并行的实现尝试或隔离实验。
- **@opencode-integrator**：opencode 集成专家。负责处理与外部命令 opencode 交互的开发、调试与测试任务，严格遵循系统的容错和数据流规范。
