# ADR-00002: 跨服务器问题数据本地缓存与落库（演进决策）

## 1. 背景与问题 (Context & Problem)
当前 `spec-00008` 的跨服务器问题列表采用**实时 Fan-out 透传**：每次请求现场并发拉取各远端 sashiko 的 `/api/patchsets` 与 `/api/patchset?id=`，在内存中映射组装为 `IssueRow` 后返回，仅辅以秒级内存短 TTL 缓存去重瞬时并发。本地 SQLite 仅持久化"服务器配置"，**不持久化问题/findings 明细**。

随着使用深入，出现新的核心诉求：

1. **跨服全量筛选与分析**：用户需要对**所有服务器聚合后的数据**进行统一筛选、排序、跨服过滤与进一步分析处理。实时 Fan-out 在"内存里临时拼出的列表"上做这些操作非常别扭，且能力受限。
2. **N+1 Fan-out 的固有延迟**：远端无扁平 findings 端点，实时模式须"列 patchset → 逐个取 findings"，耗时随服务器/问题量线性恶化。
3. **历史与离线检索**：远端节点下线后仍需查询其历史问题；实时模式做不到。
4. **降低远端压力**：实时模式下远端被请求的频率等同于用户点击频率。

因此，**本地缓存/落库是后续演进的必要能力**，但为控制 `spec-00008` 范围，本次不实现，以本 ADR 固化决策方向。

## 2. 备选方案 (Considered Options)

### 方案 A：保持实时 Fan-out + 内存短 TTL 缓存（现状）
- **机制**：不落库，每次请求实时拉取，仅内存 TTL 去重。
- **优点**：实现简单；数据永远最新；无同步/新鲜度问题。
- **缺点**：无法支撑跨服全量筛选分析；大规模下延迟抖动严重；无历史/离线检索；远端压力大。

### 方案 B：本地 SQLite 落库 + 后台定时拉取（演进目标）
- **机制**：新增本地 findings/issues 缓存表，由后台定时任务批量拉取各远端数据并落库（支持增量同步）；查询走本地 DAL，用 SQL 索引完成筛选/排序/分页/检索。
- **优点**：跨服全量筛选分析能力质变；查询性能稳定，与服务器数量解耦；支持历史/离线检索；远端压力从"每次点击"降为"每 N 分钟一次"。
- **缺点**：引入数据新鲜度滞后；需处理定时任务、增量同步、冲突与清理等复杂度。

### 方案 C：端到端真流式/消息推送同步
- **机制**：要求远端主动推送变更（webhook/消息队列）实现近实时同步。
- **优点**：新鲜度最佳。
- **缺点**：需改造 sashiko 原生代码（违反"尽量不修改原生代码"约束）；远端协作成本高。当前不可行。

## 3. 决策 (Decision)
后续演进采用 **方案 B（本地 SQLite 落库 + 后台定时拉取）**。

落地原则——**不另起炉灶，而是替换 `RemoteStatsFetcher` 的实现**：
- 复用 `spec-00007`/`spec-00008` 预留的 DAL 契约层与 `RemoteStatsFetcher` 抽象作为切换点。
- 将 `fetch_issues` 的底层数据来源从"实时 HTTP fetch"切换为"查本地落库表"，由独立的后台定时任务负责刷新落库。
- `AggregationService.fetch_issues` 的对外契约与前端 `/api/v1/issues` 保持不变，前端无感。

**本决策不进入 `spec-00008`**，待跨服全量筛选/分析诉求正式立项时由新的 Spec 承接实现。

## 4. 影响与风险缓解 (Consequences & Risk Mitigation)

1. **数据新鲜度滞后（实现时的重点考量）**
   - **影响**：落库后列表展示的是"上次同步时刻"的 triage 状态，可能滞后于远端真实状态。
   - **缓解（后续 Spec 必须设计）**：
     - **可见性**：UI 明确展示"数据更新时间 / 最近同步时刻"，让用户对新鲜度有预期。
     - **混合读取**：列表走本地缓存（重筛选/分析），点进**单条问题详情或跳转**时走实时穿透远端，保证关键操作看到最新状态。
     - **可控同步周期**：定时拉取间隔可配置；提供"手动刷新/立即同步"入口。
     - **失效标记**：远端不可达时，对应数据标注为"可能过期/最近同步于 X"，而非静默展示陈旧数据。

2. **增量同步与一致性**
   - **影响**：全量重拉在大库下不可持续；并发同步可能产生脏数据。
   - **缓解**：优先增量同步（按时间/游标）；同步任务加锁串行化；落库使用 upsert 保证幂等。

3. **存储与清理**
   - **影响**：本地缓存表随时间膨胀。
   - **缓解**：设计保留策略（如按时间窗口/容量上限清理），与"服务器配置删除时级联清理其缓存数据"。

4. **架构非侵入性**
   - 严守"尽量不修改 sashiko 原生代码"约束：同步逻辑仅消费远端既有 JSON API，全部新增代码落在 `my-src/` 下。
